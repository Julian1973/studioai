# Crystal Bears Studio — operating instructions for Claude Code

Source of truth order: CRYSTAL_BEARS_LOCKED_CANON.md (what the show is) > CRYSTAL_BEARS_STUDIO_BIBLE.md (how it is made) > this file. Where they touch, canon wins.

Every session: read this file, TICKET_PACK_001.md and the current phase of WORLD_CLASS_ROADMAP.md before writing any code. Work tickets in the pack's order of play, one commit per ticket, the commit message naming the ticket. A ticket is done only when its Definition of Done is true AND the Bible agrees — never merely when the code compiles.

Hard rules (never work around these):
1. Gates are hard locks. Never advance past an unsigned gate; never sign past an open BLOCK.
2. Taste lives in prose (the chair skills); law lives in code; canon lives in data. Anything found in the wrong layer moves the day it is found.
3. No output is ever hand patched. A fault routes through a retake with a layer diagnosis (keyframe / brief / reference / take).
4. Law 5: the voice lives in the render. One combined @Audio1, supplied by the Voice Director and lip-synced by Seedance; no native-voice fallback (a beat with dialogue whose V3 track fails REFUSES to render); no post voice swap. cb_post has no swap function by design — do not add one.
5. Never describe a character's appearance in prompt text. Identity comes only from the reference images. Binding a name to its reference slot in the prompt text ("Fuzzby is the bee from @图2") is allowed and is now the standard (Julian, 2026-07-05, the blessed v4 template — reversing this rule's earlier "names live only in the audio" wording); what stays absolutely forbidden is DESCRIBING the character — no physical/appearance text anywhere, ever. The distinction: a name is a label, not a description.
6. Canon is edited ONLY at root CRYSTAL_BEARS_LOCKED_CANON.md, then `python3 tools/sync_canon.py`. The skill copies are generated; editing one is drift. `--check` must pass before any sign off.
7. When code becomes sharper than the documents, the documents update in the same commit.
8. Restructure work follows RESTRUCTURE_SPEC_T30.md: one phase per commit, baseline byte-identical prompts proven after each phase.
9. A temporary state (e.g. a pollen moustache) resolves WITHIN the take it started in — it never carries across a take boundary (T2 ruling, 2026-07-02, Julian). There is no continuity-tail/previous-clip-tail chaining mechanism; do not re-add one without a fresh ruling.
10. cb_seedance.py's older validation machinery (the 15 minds, physical-action archetypes, authoring/compact validators) is a KEPT, intentional second validation layer underneath the Director's chair (T3 ruling, 2026-07-02, Julian) — not leftover code. It can legitimately refuse a render; that is by design.
11. When a bug is found, the fix is not done until the same pattern has been hunted across every module — not just the one file where it was noticed. State the sweep's result in the commit message: what was checked, what else was found, and what was fixed vs deliberately left for a later ticket (T33 ruling, 2026-07-02, Julian, from the startState/shotSize field-leak find).
12. A chained keyframe's environment comes SOLELY from its chain-reference image — never a competing text re-description of the scene's location alongside it. That competition is what let a chained beat (Ep1 1.B2-1.B4) drift to a visibly different world than the approved plate; `build_keyframe_prompt` fixed and locked 2026-07-02 (Julian, after Scene-1 Gate 2b review — "we need you to make sure that... is now locked in and baked into the software... a standard across the board"). Applies unconditionally, every scene, every episode — do not reintroduce a scene-level text description into a chained beat's prompt without a fresh ruling.
13. SCENE1_DIRECTORS_CUT.md is the studio's taste for comedy staging — its eleven staging laws are baked into cb_director.py's system prompt (`_mind()`, above the Pixar-masters DNA paragraph) and are HARD RULES for every comedy beat: camera-as-character, full-throttle-from-frame-one, one-gag-arc-per-clip (8-10s), simultaneous contrast (never alternating coverage), short cutaway punctuation only, never-acknowledge-failure, escalation ladder, holds-only-on-buttons (max 1, under 1.5s), one earned full stop per scene, sound-as-comic-percussion, a beat never ends on a leaf hit (the flower is always the finisher). Ep1 Scene 1 is 5 beats (1.B1-1.B5), not 4 — the old combined tumble+escalation+thunder beat was split because it violated law 3. Dialogue stays Gate-0-locked regardless of this document's own wording; reconcile and flag diffs, never silently choose (2026-07-02, Julian).

A Gate-1 deliverable change auto-relocks every downstream sign-off (cb_pipeline._relock_if_stale, mirrored in serve.py's locked_state()) — a scene's approved beats and its Gate 2/3/4/5 sign-offs are compared by content fingerprint on every gate-status read, never trusted stale. Verified by engine/test_gate_cascade.py (scratch files only); run it after touching either copy of this logic.

FRAME CHAIN doctrine (2026-07-02, Julian): a continuation beat's @图1 chains off the PREVIOUS beat's ENDING FRAME (cb_scene.build_ending_frame — the literal last frame of its rendered clip, composed as a first-class deliverable, media/<ep>_<code>_<slug>_end.png), not its opening frame — the true state after the full take beats a separately re-imagined approximation. Falls back to the opening frame when no ending frame exists yet (an unrendered beat never blocks the chain). Turnarounds (@图2/@图3+) stay in every prompt regardless — the identity anchor against chain drift. A retake upstream (a new ending frame) auto-dirties every downstream beat's keyframe+clip lock (cb_pipeline._relock_chain_if_dirty, same lazy fingerprint pattern as rule 14) — verified by the same engine/test_gate_cascade.py.

16. ACTION-STATE FIDELITY (2026-07-03, Julian — "if the first scene is them flying through the meadow, then they need to be flying through the meadow"): whenever a beat's own action text names an active locomotion verb, the KEYFRAME must show that state concretely, not just gesture at it — checked by cb_qa's new ACTION_STATE_MISMATCH (KEYFRAME_DONE.md) and generated by cb_prompts.build_keyframe_prompt's WINGS/FLIGHT-ENERGY law (asymmetric mid-downstroke wings, forward-leaning body, legs tucked — never both wings spread symmetrically with the body hanging vertical like a puppet at rest, which reads as floating in place and forces Seedance to invent a standstill-to-motion transition instead of continuing existing motion). Generation and QA ask for/verify the SAME concrete criteria, not a vague "does this look dynamic" taste call — a generic ask is what let the first real keyframes pass QA while still reading static. Deliberately excludes "hover" — the default bee state, not inherently wrong.
17. VISION-QA CHECKS MUST STATE CONCRETE VISUAL CRITERIA, NEVER SUBJECTIVE QUESTIONS (Julian's Ruling, 2026-07-03). Worked example: ACTION_STATE_MISMATCH's first draft asked the vision model a vague question — "does this pose read static, posed, standing still, or passively floating?" — and the model PASSED both of the known-bad 1.B1/1.B2 keyframes clean; it is a subjective taste call a generous grader waves through. Rewritten to ask two concrete, checkable things instead — are the wings SYMMETRICAL (fail) or ASYMMETRIC/mid-downstroke (pass)? Is the body VERTICAL with legs dangling (fail) or LEANING FORWARD into the direction of travel (pass)? — and it correctly failed both. Every vision-QA check added from here must name the specific, literally-checkable visual feature(s) that separate a pass from a fail (an angle, a count, a symmetry, a position) — never phrase the ask as an open judgment call ("does this look right/dynamic/off").
18. A NEW OR TIGHTENED QA CHECK APPLIES FORWARD ONLY, NEVER RETROACTIVELY, UNLESS JULIAN RULES OTHERWISE (his Ruling, 2026-07-03, re: ACTION_STATE_MISMATCH vs the already-signed 1.B1/1.B2). Work he has already signed off stands; a check catching something that sign-off missed is logged as calibration data (engine/QA_CALIBRATION_LOG.md) for tuning the check, never used to trigger an unrequested regeneration of signed work.
19. THE JOIN CONTRACT (JOIN_CONTRACT.md, Julian, 2026-07-03 — "the last frame needs to be the start of the next frame... we need to get it to flow across"): a handoff/edit-layer doctrine, deliberately separate from prompt-building. Every beat's final 0.5-1s is a SETTLE (character at rest in the pose that IS the next beat's startState; camera arrived at its stated end state; screen positions held at their frame-left/frame-right home); every beat's shot 1 opens FROM that settle. `cb_qa.check_join` compares beat N's composed ending frame (rule 15) to beat N+1's actual opening frame for POSITION, STATE and LIGHT — concrete criteria per rule 17, report-only. `cb_post.assemble_conformed` is Gate 4's rough-cut assembler: trims a fixed settle allowance off each clip's tail/head before hard-cutting (still no in-scene cross-dissolves) so a cut lands mid-motion instead of hold-into-hold; `assemble_picture` (the raw butt-join) stays, unchanged, as the deliberate comparison baseline. Diagnosing a join is not license to touch prompt code or re-render — a join found broken only names what's wrong; the fix (if any) is a separate, later ruling.

THE HANDLE DOCTRINE (Julian, 2026-07-03 — "every shot shoots long so the cutter has meat to trim into"): every beat now renders at 15s (`cb_segprompt.HANDLE_TOTAL`), superseding the old per-beat 8-15s `durationSec` range. 13s (`HANDLE_ACTION`) is the story-action budget, split across the beat's own shots by weight exactly as before; the final 2s (`HANDLE_SETTLE`) are appended to the closing shot as a DIRECTED LIVING SETTLE — never dead air, per the dead-time-causes-hallucinations rule — built from the beat's own `endState` field (`cb_segprompt._v3_settle`; a universal idle-life guarantee is always appended even before every beat has an authored `endState`). The settle window serves two jobs at once: the relay's harvest window (rule 21 — the sharpest frame ANYWHERE in those 2s, not a prayer that the literal last frame is clean) and Gate 4's trim handle (rule 19 — the editor cuts into the settle per join). Both emitters carry it; goldens re-blessed same day.
21. THE RELAY CHAIN (Julian, 2026-07-03): a beat's OWN Gate-2b keyframe is generated only when it has no signed predecessor to open from. A continuation beat instead opens directly off the previous beat's signed clip's HARVESTED SETTLE FRAME (`cb_scene.harvest_settle_frame` — the sharpest frame anywhere in its 2s settle window, scored by Laplacian variance, never just the literal last frame), RE-MINTED per rule 22 before it anchors anything, fed into Seedance's ref2vid as `@图1` — no separate keyframe-image-generation step for that beat at all. This finally applies the FRAME CHAIN doctrine's pending correction (rule 15's note, LAB_BACKLOG.md's "Chain doctrine note"): an ending frame is a render-time first-frame carry, never a generation anchor — resolved by removing the anchor step entirely for relay beats rather than reverting to opening-frame chaining. A 4th reference (the scene's plate) anchors the world's canonical look/palette without forcing the frame, alongside `@图1` and the character turnarounds. Applies to a beat only once its predecessor has an actual SIGNED clip — an unsigned/undecided predecessor blocks the relay at that beat, it does not fall back to guessing.

THE RE-MINT STEP (Julian's ruling, 2026-07-03 — "the relay gains the RE-MINT step as standard... re-mint is now every link, not QA triggered"): supersedes the earlier same-day lab rejection of a "NB2 chain refresh" (LAB_BACKLOG.md) — the director overrides the lab, explicitly and on the record. Every harvested settle frame is re-minted through NB2 with a LOCKED, deliberately minimal prompt (`cb_prompts.build_remint_prompt`) before it may anchor the next beat: turnarounds attached solely to hold identity, artifacts and motion blur removed, nothing else changes — a restoration pass, never a re-generation. `cb_qa.check_remint` is a hard BLOCK comparing the re-mint against the harvest (position/state) and the turnarounds (identity). Julian approves the cleaned anchor before it launches anything: `cb_beats.fire_next_beat` prepares (designate → harvest → re-mint → drift-check) and STOPS, returning only once called again with `approved=True` — no auto-advance past an unapproved anchor, ever. `cb_scene.relay_source_for` prefers an already-produced re-mint over the raw harvest, falling back to the raw harvest only if a relay beat is fired outside the `fire_next_beat` ceremony (graceful degradation, not a silent skip of the step when it's actually available).

24. THE FOUR-ANCHOR GROUNDING SET (Julian, 2026-07-04 — "point anchor, duration anchor, identity anchor, performance track. That's the full grounding set — nothing left for the model to invent."): every relay beat's reference stack names FOUR anchors, each doing exactly one job, stated as such in the prompt text itself so the split is never left implicit. POINT anchor (@图1, the harvested/re-minted settle frame): this instant only — "this image IS the first frame... nothing is composed fresh" — now also carries a CONTENT description mechanically derived from the previous beat's own authored `endState` field (`cb_segprompt._v3_prev_frame_content`), so the model is told what @图1 shows, not just its job. DURATION anchor (the scene plate, always the last @图N): the environmental constant for the WHOLE clip — lighting direction, palette, textures — "the world must not drift from this plate," explicitly never a framing/pose reference. IDENTITY anchor (the character turnarounds, @图2/@图3...): proportions, colours, features, held every beat. PERFORMANCE track (@Audio1): mouth, timing AND full physical performance, camera locked whenever any character speaks. All four are mechanical — derived from existing beat/scene data, nothing invented — and generalize forward automatically: once a beat is relayed, its own `endState` becomes the NEXT beat's point-anchor content, with no beat-specific code (verified live for 1.B3 ahead of its own re-fire). THE LOCKED RELAY-BEAT TEMPLATE (Julian, 2026-07-04 — "This is the locked relay-beat template — canonical for every beat after a scene's opening anchor, all scenes, from now."): the JSON emitter's (`emit_json_v3`) shape is now the SCHEMA every relay beat's shipped prompt follows, every scene, every episode. Fields and their jobs:
    - `references`: the four-anchor grounding set (rule 24) — one entry per @图N, size-only identity handles (below).
    - `voices`: the PERFORMANCE track law — @Audio1 only, full physical performance, camera locked while any character speaks (rule 24).
    - `world`: the actual environment/atmosphere prose, each element stated EXACTLY ONCE — never repeating what `references` already declared for @图1/the plate (the duplication-with-a-broken-join bug found and fixed 2026-07-04: `_v3_environment`'s `include_refs=False` mode for this field).
    - `tone`: the beat's emotional register, mechanically mapped from `comedyMode` (`_v3_tone`).
    - `ambience`: THE LOCKED AMBIENT BED — word-for-word identical every clip in the scene, read straight from the scene's own `ambientBed` field (`_v3_ambience`), never varying beat to beat; each beat's own `soundIntent` supplies foreground SFX layered ON TOP, per shot — the bed is the constant, the foreground is what changes.
    - `rule`: the written invariant for the whole beat (wing law, camera-lock-during-dialogue, hold/timing-lock) — never per-shot, always whole-beat.
    - `shots`: the per-shot camera/action/dialogue breakdown. Action text keeps its own established "full descriptive label on first mention, short label after" cadence (`_delabel`) — that stays exactly as is; it is PERFORMANCE DIRECTION, a different job from the binding handle below, and Julian confirmed it explicitly: "leave the action-prose flavour language as-is."
    - `constraints`: one positive-framed identity-consistency line.
    - `negatives`: SIX standing items, always exactly six. FOUR are universal, every scene, every cast: no morphing/redesign/rescale of the characters; no extra characters or props; no on-screen text/subtitles/logos/watermarks; no foreign-language speech. The remaining TWO are species-appropriate and swap by cast (`_v3_negatives`'s `any_bee` branch) — for a bee scene: no crystals on/attached to the bees, no still/frozen wings while airborne; for a non-bee cast: no flicker/compression artifacts, no continuity break in lighting/palette. A future scene's cast decides which pair applies — never invent a seventh.
    Binding handles (the SUBJECTS line and each character's `references` entry) are SIZE-ONLY ("the larger bee") — personality adjectives ("eager, manic") were trimmed out of the identity-binding declaration specifically (Julian: "trim the personality adjectives from the binding handles — size handles only"), since that label's only job is binding an @图N slot to a character, not directing performance.

THE TEMPORAL-CONTRADICTION BUG — `endStateStill`, a field separate from `endState` (Julian, 2026-07-04, diagnosing the very next fire under rule 26's fix, which STILL failed the join check the same way): three errors in one field, all traced to `_v3_prev_frame_content` injecting the previous beat's `endState` VERBATIM as @图1's content description. `endState` is DIRECTING PROSE for that beat's own SETTLE block — full of temporal verbs and imperatives ("straightens into... and holds it", "End on his pose... both at rest, camera locked, as the meadow's hum resumes"). Pasted into @图1's description for the NEXT beat, Seedance has no way to know those verbs describe yesterday: "End on his pose, both at rest" reads as THIS clip's own ending instruction, directly contradicting rule 26's own anti-hold clause ("by the shot's end the characters are in a completely different position") — two opposing ending instructions in one prompt, and the render satisfied the wrong one, holding the anchor pose for the whole clip exactly as before. It also near-duplicated shot 4's own SETTLE wording, and restated the `ambience` field a second time, wrongly bound to the opening reference.

THE PRODUCTION DOCTRINE — THE WHOLE MACHINE, ONE PAGE (Julian, 2026-07-04 — "once all twelve are code-enforced, every prompt for the next ten scenes is built by the same hands, and the only thing you ever author again is Layer 2, which is the job you actually wanted: directing"; CONSOLIDATED 2026-07-05, lock-in night, absorbing rules 29-36 into a single standalone, current statement). This rule absorbs all of it — the numbered rules stay in place as the dated record of WHY each change happened; this rule is WHAT IS TRUE NOW, the one to check against going forward. The same doctrine also lives at repo root as `PRODUCTION_DOCTRINE.md` (the one document a new session reads first) and in `REPLICATOR.md`'s own gate map — all three describe the same machine; if they ever disagree, fix the disagreement the day it's found (rule 7).

    THE HIERARCHY: Episode → Scene → Beat.

    THE SCENE BUBBLE (rule 35): a scene's identity is its THREE LOCKED CONSTANTS — the scene plate, the ambient bed, the style law — held VERBATIM across every beat inside it. A scene boundary is a FULL, DELIBERATE RESET: new plate, new ambient bed, a fresh canon-generated anchor keyframe for the scene's own opening beat, relay depth back to zero, no visual continuity owed across the boundary. Every law below applies identically inside every bubble.

    THE BEAT: one gag arc, 15 seconds hard-locked — 13s action + 2s settle (the Handle Doctrine, rule 20). Never two arcs in one take.

    LAYER 1 — THE INVARIANT SKELETON (machine-filled, identical logic every prompt; a law here can be forgotten only if it's convention rather than code — see the audit this rule commissioned, PROMPT_LAWS_AUDIT.md):
    1. One beat, one prompt, one gag arc. 15s = 13s action + 2s settle. Never two arcs in one take.
    2. Opener vs relay vs junction type: a scene's first beat opens from its generated canon anchor keyframe. Every beat after declares its OWN junction type (a beat-level field, `junctionType`, rule 31) — `intentional_next_shot` (THE DEFAULT: beats join by CUT, not freeze — a new gag arc, a fresh camera setup, energy already in frame) or `seamless_continuation` (ONLY where the director's own cut explicitly continues one shot across the boundary) — and opens from the previous beat's harvested settle frame either way. Omission is NEVER read as seamless — that stays the rarer, explicitly-declared case.
    3. The reference stack, every relay prompt, each entry one declared job: @图1 — for a `seamless_continuation` beat, the locked opening frame (point anchor); for an `intentional_next_shot` beat (the default), a STATE REFERENCE only — identity, declared carry marks, lighting and world-position carry forward exactly; camera, pose, energy and motion do not (rules 31/33/34). @Video1 — motion energy and action continuity ONLY; never camera framing, shot size or composition, which comes from THIS beat's own direction (the corrected line, FIXED 2026-07-05 lock-in night — the earlier wording, "previous visual motion continuity and energy," still left framing implicitly copyable). Character turnarounds — identity anchor, one per character, held every beat. Scene plate — duration anchor, the environmental constant for the whole clip (lighting, palette, texture), never a framing or pose reference. @Audio1 — the sole vocal source, covering lines, hums and sing-song rhythm alike; performance anchor: timing, mouth rhythm and full physical performance, camera locked whenever any character speaks.
    4. The @图1 description is a photograph for a `seamless_continuation` beat (content, not choreography — no temporal verbs, no imperatives, no "end on") or the fixed state-carry template for an `intentional_next_shot` beat (Law 3) — never a mix of the two, and never the predecessor's full endState sentence stuffed in wholesale (rule 33's rewrite of `_v4_state_carry`).
    5. Anti-hold — `seamless_continuation` ONLY: frame one IS @图1 exactly, then motion moves decisively away, immediately; never held, repeated or returned to (rule 26). An `intentional_next_shot` beat (the default, rule 31) never opens on @图1's frame at all — it opens on its own fresh camera setup, carrying only @图1's declared state forward; the model is never asked to hold or repeat a pose that isn't this beat's own.
    6. Audio law: all vocal sound — lines, hums, exclamations, sing-song — is the V3 performance fired in as @Audio1. Seedance generates no voice-like sound of its own. The prompt never contains the spoken words — dialogue binds speaker, timing and expression only, by name and ordinal.
    7. Ambience is scene-property: one locked ambient-bed line, word-for-word identical across every prompt in the scene (the Scene Bubble Law). Per-beat SFX layers on top; ambience never appears inside any other field.
    8. Camera law: locked during any vocal beat; motion only in non-speaking action; species-scaled (mechanical, from the cast — "bee-scale") and naming the SPECIFIC eyeline/action characters when the beat has declared an `opensOn` pair (rule 33), falling back to generic phrasing otherwise.
    9. The living settle: every clip ends on ITS OWN completed beat, in character — read from the beat's own authored `endState` VERBATIM (rule 33; never the old generic idle-life placeholder once `endState` is authored), so a manic character stays visibly manic at rest and a deadpan character gets its own full pause. NEVER a restatement of the previous beat's pose — the settle is this beat's OWN distinct moment (advisory-checked by `check_settle_distinctiveness`); that settle is the next beat's harvest.
    10. The six negatives, every prompt, in full — Crystal World Rule and Wing Law always — plus the beat's own `stagingProhibited` items additive on top (rule 30), never fewer than six standing.
    11. Character law: identity by reference only, name-welded directly to its @图N slot ("Fuzzby is the bee from @图2," rule 5/30) — a name is a label, never a description; no physical/appearance text anywhere, ever.
    12. Style law and world text: verbatim from the show profile and scene data, never retyped, never paraphrased.

    LAYER 2 — THE CREATIVE SLOTS (director-authored; the ONLY things that change beat to beat):
    1. The beat's own junction type (Law 2) — defaults to cut; declare `seamless_continuation` only when the shot genuinely doesn't travel.
    2. Action prose: the beat's own gag, ERRATIC IN CHARACTER, PRECISE IN CHOREOGRAPHY (rule 33) — every manic action a SPECIFIC NAMED GAG with cause and consequence (rockets, brakes too late, loops once, stops); adjective-chaos ("wildly," "crazily," a generic frenzy word with no physical beat behind it) is BANNED as unreadable — the model executes actions and butchers adjectives.
    3. `opensOn: {who, action}` — WHO the camera opens on and their immediate mid-motion state; the Coverage Law's reaction-bridge mechanism, rendered as a fixed "OPEN ON {NAME} mid-motion: {action}." sentence — a foil character's look or turn (Zenny's head-turn) is the house bridge whenever the cut travels.
    4. `carryMarks` — a SHORT phrase (never a full sentence) naming what specifically, visibly persists (a moustache, leg pollen) — the ONLY thing the join-check's STATE tier hard-gates on (rule 36), and what the NEXT beat's own @图1 clause quotes.
    5. `endState` — directing prose for THIS beat's own ending, written as its own distinct moment in character, never a restatement of the predecessor's pose. `endStateStill` — the same moment as a static photograph, hand-authored in parallel (rule 27), never auto-derived from `endState`.
    6. `actingRule` (per character, a new one-line `characters.json` bible field) + `actingContrast` (per beat, which characters play off each other and how) — feed the `acting_rules` field.
    7. Dialogue bindings (who speaks, expression direction), tone, per-beat foreground SFX, `stagingProhibited` (beat-specific failure modes, merged with the six standing negatives).

Every beat opens on a NEW ANGLE WITHIN the same space, close to where the previous beat ended, motivated by eyeline or motion (Layer 2.3's `opensOn`) — never a relocation, never a fresh establishing wide mid-scene (the wide belongs to the scene's opening beat only). Distinct from GEOGRAPHY (which only asks "is this the same world at all") — a technically-same-location shot can still fail COVERAGE by reading as a re-establish.
